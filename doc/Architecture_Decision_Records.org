* DRAFT Normalize input text and translation rules to a canonical Unicode form
- Deciders :: CE
- Date :: [2026-06-23 Di]

** Context and Problem Statement

Unicode text can represent the same character in multiple equivalent
forms. For example, the Portuguese letter ~à~ can be encoded as a
single precomposed code point (U+00E0, NFC form) or as a base letter
~a~ followed by a combining grave accent (U+0061 U+0300, NFD form).
When input and table rules use different forms, translation fails:
the rule for the precomposed character does not match the decomposed
input, producing garbage output.

This problem is tracked in [[https://github.com/liblouis/liblouis/issues/98][liblouis issue #98]] and has been known since
at least 2015.

A first attempt at a solution normalized input to NFC at the pipeline
boundary. This fixed the Portuguese grade-1 table (pt-pt-g1.utb),
which defines rules for precomposed characters. However, it broke
tables that deliberately define rules for standalone combining
characters:

- IPA (~IPA-unicode-range.uti~) defines ~sign \x0302 4-146~ (combining
  circumflex = falling tone mark). After NFC, ~e + U+0302~ becomes
  ~ê~ (U+00EA); the combining-mark rule can never fire.
- Similarly for Hebrew (~hbo.yaml~), Yiddish (~yi.yaml~), and Greek
  (~el-forward.yaml~).

The Portuguese table alone needed 63 explicit ~correct~ rules for
NFD → NFC conversion in its computer-braille variant; the grade-1
variant had none and was silently broken for NFD input.

** Decision Drivers

- Table authors should write rules in a single, canonical form and not
  need to worry about which normalization form their input will arrive in
- Users should not need to know about NFC vs NFD when providing input
- louis-rs aims for a cleaner, more predictable contract than liblouis
- The solution must work for both classes of table:
  1. Tables that define rules for precomposed characters (e.g. ~letter á~)
  2. Tables that define rules for standalone combining characters (e.g. ~sign \x0302~)
- Braille codepoints (U+2800–U+28FF) have no canonical decompositions,
  so the choice of normalization form is a no-op for backward translation

** Considered Options

1. *NFC (Canonical Composition)*: normalize
   both input and rules to NFC. Precomposed forms win. Fixes class-1
   tables (precomposed rules, NFD input), breaks class-2 tables
   (standalone combining-mark rules).

2. *NFD (Canonical Decomposition)*: normalize both input and rules
   to NFD. Every precomposed character is decomposed into base + combining
   sequence. A table rule for ~á~ (U+00E1) is stored in the trie as the
   two-character path ~a + U+0301~; input ~á~ or ~a + U+0301~ both
   decompose to ~a + U+0301~ and follow the same path. A table rule for
   standalone ~\x0302~ is already in NFD form (a lone combining mark has
   no canonical decomposition). Input ~e + U+0302~ stays as-is; ~e~
   matches the ~e~ rule, ~U+0302~ matches the combining-mark rule. Input
   ~ê~ decomposes to ~e + U+0302~ and is handled the same way.

   NFD therefore appears to satisfy both classes of table simultaneously.
   It has not yet been implemented and corner cases (word-boundary
   detection across grapheme-cluster splits, case-insensitive matching
   with decomposed characters) have not been fully evaluated.

3. *Leave normalization to the caller* (the liblouis approach):
   no normalization at the library level. Table authors add explicit
   ~correct~ rules for each NFD sequence they expect. This is verbose
   (63 rules for one Portuguese variant) and error-prone — tables that
   omit the rules silently produce garbage output.

4. *Per-table opt-in*: a table metadata directive (~#-unicode-form:~)
   selects the normalization form per table. Flexible but shifts
   complexity to table authors.

** Decision Outcome

Currently implemented: option 1 (NFC) — provisionally, pending
investigation of option 2 (NFD).

Both input (~pipeline.rs~) and multi-character rule patterns (~trie.rs~)
are normalized to NFC. Translation rules that define standalone combining
characters break silently under NFC; a compile-time warning should be
added when a rule pattern changes form after normalization (marked TODO
in ~trie.rs~).

We specifically use NFC (not NFKC). NFKC performs additional
compatibility decompositions (e.g. ligature ~ĳ~ → ~ij~) that change
character counts more aggressively and are outside the scope of what
table authors expect.

Liblouis chose option 3 for two reasons: the original author wanted
liblouis to be self-contained with no external dependencies, using gnulib
for utility code; and normalization changes character counts for decomposed
input (~a\x0300~ = 2 chars → ~à~ = 1 char), which breaks cursor-position
mappings. In Rust, the ~unicode-normalization~ crate makes this trivial,
and louis-rs does not currently implement position mapping.

** Positive Consequences

- Users can supply text in any Unicode normalization form
- Table authors write rules in a single canonical form
- Normalization ~correct~ rules in tables become unnecessary
- The ~#-unicode-form:nfc~ metadata hint becomes unnecessary

** Negative Consequences (current NFC implementation)

- Translation rules that explicitly target isolated combining diacritics
  will never fire. NFC canonical ordering reorders sequences of combining
  marks by their Canonical Combining Class values, so input that was
  written expecting a specific ordering is silently mis-translated.
  Measured impact against the liblouis test suite (713,827 tests total):

  | Table                | Before NFC | With NFC | Change | Tests |
  |----------------------+------------+----------+--------+-------|
  | ~ipa.yaml~           |     100.0% |    76.5% | −23.5% |    34 |
  | ~yi.yaml~            |      93.8% |    68.8% | −25.0% |    16 |
  | ~ar-ar-comp8.yaml~   |     100.0% |    95.8% |  −4.2% |    24 |
  | ~el-forward.yaml~    |      84.9% |    82.8% |  −2.1% |    93 |
  | ~hbo.yaml~           |      43.7% |    41.8% |  −1.9% |   469 |
  | ~bn.yaml~            |      36.8% |    35.6% |  −1.2% |    87 |
  | ~vi.yaml~ (improved) |      47.6% |    53.7% |  +6.1% |    82 |
  | ~pt.yaml~ (improved) |      84.8% |    87.0% |  +2.2% | 1163  |

  NFC is a net gain on the weighted total (~pt.yaml~ alone has 1163
  tests), but it systematically breaks every table that uses standalone
  combining marks.

- No warning is currently emitted when a combining-mark rule is silently
  unreachable (tracked as TODO in ~trie.rs~).

** Future Considerations

- Evaluate NFD as a drop-in replacement for NFC in both ~pipeline.rs~ and
  ~trie.rs~. Two open issues must be resolved first:

  1. *Word-boundary detection at decomposed-sequence seams.* Under NFD,
     a precomposed letter like ~á~ (U+00E1) is split into two codepoints:
     base ~a~ + combining mark U+0301. After the trie consumes both
     codepoints and returns the translation, the pipeline's ~prev~ pointer
     is left at U+0301 — a combining mark, which is not a letter. When
     the next character (say ~r~ in "árbol") is looked up, any
     ~Start(Word)~ boundary check sees ~prev~ = U+0301 (non-letter) and
     ~c~ = ~r~ (letter), and incorrectly fires a word-boundary in the
     middle of the word. Under NFC, ~prev~ after translating ~á~ is the
     precomposed letter ~á~ itself, so no false boundary fires. The fix
     is to advance ~prev~ by grapheme cluster rather than by codepoint,
     but this logic does not currently exist.

  2. *Case-insensitive trie matching with decomposed uppercase.* The trie
     normalises character transitions to lowercase via ~char::to_lowercase~.
     Under NFD, uppercase ~Á~ decomposes to ~A~ + U+0301. ~A.to_lowercase()~
     = ~a~, so the base is handled correctly, but it is unclear whether
     multi-codepoint uppercase sequences (some scripts produce more than one
     codepoint from ~to_lowercase~) are handled correctly in all cases.
- If position mapping (cursors/typeforms) is added, the normalization step
  must also produce a character-index mapping so callers can relate
  positions in the original input to positions in the normalized input.
  NVDA solved this (see [[https://github.com/nvaccess/nvda/pull/16521][nvaccess/nvda#16521]]).

* ACCEPTED Pre-compute indication events before translation
- Deciders :: CE
- Date :: [2026-06-23 Di]
- Implemented :: [2026-06-26 Do]

** Context and Problem Statement

Braille indication — the insertion of indicator cells to signal
numeric sequences, uppercase runs, letter signs, and emphasis — is
currently implemented as a set of mutable state machines
(~numeric::Indicator~, ~uppercase::Indicator~, ~lettersign::Indicator~,
~nocontract::Indicator~) that walk the input in parallel with the
translation loop. At each position the translation loop calls
~indicators.next()~, which advances all state machines and returns any
indication translations to emit.

This works for built-in indication, but two problems remain unsolved:

1. *Typeform integration.* Users can supply a per-character ~typeforms~
   array annotating emphasis (italic, bold, …). These annotations also
   require start/end indicators in the braille output, but the typeform
   array is entirely separate from the indicator state machines. There
   is no clean mechanism to merge the two sources of indication.

2. *Awkward mutable clone.* The translation table (~PrimaryTable~) is
   intended to be immutable and shared, but the indicators carry mutable
   state, so they must be cloned for every translation call (see the
   FIXME at ~primary.rs:722~). This is a structural smell.

The question is whether indication events can be *pre-computed* from
the input text before the translation loop begins, producing an
immutable per-position event array that both the built-in state
machines and the typeform array populate, and that the translation loop
consults by simple indexed lookup.

*** Does indicator state affect rule selection?

This is the key correctness concern: if which rules fire depends on
indicator state, pre-computing events before translation may produce a
different answer than computing them on the fly.

Investigation of the liblouis C source (~lou_translateString.c~) reveals
that indicator state *does* influence rule selection via a ~dontContract~
flag:

- ~checkNumericMode()~ is called at every position before
  ~for_selectRule()~. When it detects entry into numeric mode (a digit
  or ~numericmodechars~ character followed by a digit) it sets both
  ~numericMode = 1~ and, when the table's ~noContractSign~ equals its
  ~noNumberSign~, ~dontContract = 1~.
- ~dontContract~ is passed into ~for_selectRule()~ and suppresses about
  15 contraction-type rules (~CTO_Always~, ~CTO_WholeWord~,
  ~CTO_BegWord~, ~CTO_EndWord~, ~CTO_Match~, and others).
- The ~no_contract~ bit in the user-supplied ~typebuf~ (typeforms) also
  sets ~dontContract~ directly (~lou_translateString.c:3684~).

Crucially, both the indication emission (numsign) and the
contraction-suppression (~dontContract~) are computed from the *same
source*: character class membership of the input characters (~CTC_Digit~,
~CTC_LitDigit~, ~CTC_NumericMode~, ~CTC_NumericNoContract~). Neither
depends on what braille has already been output. The two effects happen
together as consequences of the same input-character analysis, not as a
causal chain where emitting the indicator changes subsequent rule
selection.

Therefore both the indication events and the ~dontContract~ spans are
*pre-computable* from the input text and the character class
definitions.

*** The ~prevTransOpcode~ / ~exactdots~ edge case

~for_selectRule()~ also checks ~prevTransOpcode != CTO_ExactDots~ before
accepting ~CTO_MidNum~ and ~CTO_EndNum~ rules. This looks like a genuine
translation-decision dependency. However:

- ~exactdots~ is an opcode intended exclusively for
  [[https://github.com/liblouis/liblouisutdml][liblouisutdml]] semantic-action files, which inject raw dot patterns
  into a translation mid-stream via ~@dots~ sequences in the source
  document.
- No standard braille table shipped with liblouis uses ~exactdots~.
- Louis-rs operates on plain text input with table-driven translation;
  there is no mechanism to inject raw dot patterns mid-stream.

The ~prevTransOpcode == CTO_ExactDots~ guard is therefore outside the
scope of louis-rs and can be disregarded for this decision.

** Decision Drivers

- Built-in indication (numeric, uppercase, lettersign, nocontract) and
  user-supplied typeform indication are the same phenomenon and should
  be handled by the same mechanism
- The mutable-clone pattern for indicators should be eliminated
- The translation loop should be simplified: indication should not
  require threaded mutable state alongside rule selection
- Contraction suppression in numeric mode (~dontContract~) must remain
  correct under the new design
- The data structure should be cache-friendly (no heap allocation per
  character position)

** Considered Options

1. *Status quo with typeform state machine added* — keep the existing
   mutable state machines, add a parallel state machine over the
   typeform array, merge outputs at each position in the loop.

2. *Pre-computed events array* — run all state machines (and the
   typeform array) as a single preprocessing pass before translation
   begins, producing an immutable ~Vec<IndicationFlags>~ indexed by
   character position. The translation loop consults this array by index.

** Decision Outcome

Chosen option: *Pre-computed events array* (option 2).

The preprocessing pass runs each indicator state machine and the
typeform array once over the input string, recording at each character
position a bitflag word that encodes:

- Which indication events fire at this position (emit numsign, emit
  capsletter, emit lettersign, emit emphasis start/end, …)
- Whether contraction is suppressed at this position (~dontContract~
  equivalent, derived from numeric mode and the ~no_contract~ typeform
  bit)

The translation loop replaces ~indicators.next(chars.as_str(), prev)~
with a single array lookup. The indicators no longer need to be cloned;
the array is immutable and cheaply shareable.

The event type is defined in ~src/translator/indication/events.rs~.
Each position uses ~EnumSet<IndicationEvent>~ (from the ~enumset~ crate),
which compiles to a compact integer with no per-position heap allocation.
~Indicators::precompute()~ returns a ~PrecomputedIndications~ value that
bundles the per-position event array with the event-to-translation map;
callers ask for ~translations_at(pos)~ directly.

** Positive Consequences

- Typeforms and built-in indication are unified: both populate the same
  array during preprocessing, both are consumed identically in the
  translation loop.
- The mutable-clone FIXME in ~primary.rs~ is eliminated; ~PrimaryTable~
  becomes fully immutable.
- The translation loop is simpler: one array lookup per position instead
  of calls into multiple state machines.
- Indication logic is testable in isolation: the preprocessing pass can
  be unit-tested independently of the translation loop.
- Contraction suppression (~dontContract~) is also pre-computed,
  keeping the liblouis-equivalent behavior correct without live state in
  the loop.

** Negative Consequences

- An additional allocation proportional to input length is required
  before translation begins. For typical inputs this is negligible.
- The preprocessing pass is a second traversal of the input. This is
  also negligible compared to the translation work, but it is a
  structural change that new contributors will need to understand.
- If a future liblouis feature introduces genuine feedback from
  translation decisions back into indication state (analogous to
  ~prevTransOpcode~ for ~midnum~, but without the ~exactdots~ escape
  hatch), the pre-computation assumption would need to be revisited.

** Pros and Cons of the Options

*** Status quo with typeform state machine added

- Good, because it is a minimal change and does not restructure the
  translation loop
- Bad, because it adds yet another parallel mutable iterator to an
  already crowded loop
- Bad, because built-in and typeform indication remain structurally
  separate with no shared abstraction
- Bad, because the mutable-clone pattern remains

*** Pre-computed events array

- Good, because it unifies built-in and typeform indication under one
  abstraction
- Good, because it eliminates mutable state from the translation hot loop
- Good, because the preprocessing step is independently testable
- Good, because the bitflag representation is cache-friendly with no
  per-position heap allocation
- Bad, because it requires a dedicated preprocessing pass and the
  associated ~Vec~ allocation

- ~src/translator/indication.rs~ — ~PrecomputedIndications~, ~Indicators::precompute()~, ~From<TextAttribute>~ and ~From<&[TextAttributes]>~ bridge impls
- ~src/translator/indication/numeric.rs~ — numeric indicator ~precompute()~
- ~src/translator/indication/uppercase.rs~ — uppercase indicator ~precompute()~

  ~checkNumericMode~, ~dontContract~, and ~for_selectRule~

* ACCEPTED Backward-translation indication with pre-computed events
- Deciders :: CE
- Date :: [2026-06-26 Do]

** Context and Problem Statement

Forward-translation indication is now pre-computed: each indicator runs
a single pass over the text input and records, at each character
position, which events fire there. The translation loop consults an
immutable ~PrecomputedIndications~ array instead of advancing mutable
state machines in parallel.

Backward translation (braille → text) poses the symmetric but inverted
problem. In forward translation the indicators are *inserted* — the
state machines detect conditions in the text and emit braille indicator
cells. In backward translation the indicator cells are already present
in the braille input and must be *detected* and *consumed*, with their
meaning applied to how subsequent braille cells are interpreted.

The question is how to extend the pre-computed architecture to cover
backward indication, and in particular whether the same ~precompute()~
interface and ~PrecomputedIndications~ type can be reused or whether a
separate abstraction is needed.

*** Detection vs insertion

In forward translation, ~numsign~ means: /when the input text enters a
numeric run, emit the numsign dot pattern before the first digit./
In backward translation, ~numsign~ means: /when the braille input
contains the numsign dot pattern, the following cells represent digits./

The indicator cell itself has no text output — it is consumed and
suppressed — but it sets a mode that affects how subsequent braille
cells are matched against back-translation rules. This is analogous to
the forward ~DontContract~ flag, but inverted: instead of suppressing
contractions in the *source*, it forces digit interpretation in the
*target*.

*** Same opcode, inverted state machine

The same table opcode (~numsign~, ~capsletter~, ~begcapsword~, …) drives
both directions. In forward translation the state machine watches for
triggering characters in the text and emits the indicator cell. In
backward translation the state machine watches for the indicator dot
pattern in the braille input and sets a mode flag for subsequent cells.

The key question for pre-computation: can the backward state machine
also analyse the entire braille input in one pass before the translation
loop, producing a per-position array of interpretation flags?

*** Does the output affect subsequent indicator detection?

In forward translation, pre-computation is valid because the indicator
events depend only on the input text characters, not on what braille has
already been emitted. The analogous question for backward translation is
whether the detection of indicator cells in the braille input depends on
what text has already been output.

For numeric indication this is straightforward: the numsign dot pattern
is a fixed, recognisable cell. Its detection depends only on the braille
input sequence, not on the output text produced so far.

For uppercase indication the picture is similar: ~capsletter~ and
~begcapsword~ are fixed dot patterns. Their detection is independent of
prior output.

However, unlike forward translation where we know the exact character
boundaries of the input text, braille back-translation operates on a
sequence of braille cells whose correspondence to output characters is
determined by the rule matching. This means a braille-position index is
not the same as a text-character index, and the two cannot be trivially
unified.

*** Implication for rule selection

In forward translation, the ~DontContract~ event pre-computed at a text
position suppresses contraction rules when that position is translated.
In backward translation, a detected ~numsign~ suppresses non-digit
back-translation rules for the following braille cells. The mechanism
is the same in structure — a per-position flag that modulates rule
selection — but the positions are braille-cell positions rather than
text-character positions.

** Decision Drivers

- The pre-computed architecture should extend naturally to backward
  translation rather than reverting to the mutable-state-machine pattern
- The correspondence between braille-cell positions and output-text
  positions must be handled consistently
- The same ~IndicationEvent~ vocabulary should be reusable where possible
- Backward indicator detection must remain correct even for tables that
  define multi-cell indicator patterns

** Considered Options

1. *Separate backward precompute pass over braille input* — analogous to
   the forward pass but running over braille cells. Produces a
   ~PrecomputedIndications~ indexed by braille-cell position. The
   backward translation loop consults it by braille-cell index.

2. *Inline detection during backward translation* — detect indicator
   cells opportunistically as the backward loop encounters them, without
   a separate pass. Avoids the braille-position / text-position alignment
   problem but reintroduces mutable state in the translation loop.

3. *Reuse forward precompute on the back-translated output* — run the
   forward indicator state machines on the *reconstructed text* after
   a first-pass back-translation, then use the results to refine a
   second pass. Correct in principle but doubles the translation work
   and introduces a two-pass architecture.

** Decision Outcome

Chosen option: *Separate backward precompute pass over braille input*
(option 1).

The alignment worry — that braille-cell positions and output-character
positions don't correspond one-to-one — turned out not to block
pre-computation, because detection only ever needs the braille-cell
index: the pass walks the braille input once and records, per cell,
which interpretation constraints apply (force uppercase, force digit
reading, etc.). The translation loop still resolves cells to text via
normal rule matching; it just consults the constraint at each cell's
position along the way, the same way forward translation consults its
precomputed events at each text position.

In the process this generalized further than either ADR anticipated:
forward and backward indication now share a single per-position
constraint mechanism, rather than a ~PrecomputedIndications~-shaped
type on one side and something bespoke on the other. Numeric-mode
suppression, computer-braille spans, and the backward caps/digit
disambiguation this ADR was written for are all instances of the same
idea — a set of constraints attached to a position, computed once
before translation, consulted read-only during it.

* SUPERSEDED Regexps duplicate character transitions
- Deciders :: CE
- Date :: [2024-12-24 Di]

** Context and Problem Statement
We have a graph for all regular expressions. For alternatives we
create a new node and add all the alternatives between the current and
the new node. Now what do we do if there is already a character
transition for one of the characters in the alternatives? We could
either
1. merge the nodes
   - how do we handle the case where both nodes are accepting, i.e.
     have translations? Which translation gets precedence?
2. add an epsilon transition
   - from the new node to the existing one
   - and delete the edge
3. allow multiple character transition edges with the same character
   from the same node
   - this of course totally breaks the current data structure

#+begin_example
letter a 1
letter b 2
letter o 3
letter r 4
letter z 5

always bor 17
match b(a|o) r z 14
#+end_example

#+begin_example
/bor|b(a|o)rz/
#+end_example

** Decision Drivers
** Considered Options
** Decision Outcome

Superseded — the question is moot. The regex engine was later
rewritten from a shared character-transition graph to a VM that
compiles alternation to a pair of branching instructions, each
alternative getting its own instruction stream rather than a node
other patterns might also land on. Matching runs all live branches
in parallel and asks which one finished, so two alternatives that
happen to start with the same character never need to merge, share,
or otherwise coordinate — there is no shared node for them to
collide on in the first place.

* DRAFT Hyphenation
- Deciders :: CE
- Date :: [2024-12-20 Fr]

** Context and Problem Statement

Liblouis uses hyphenation dictionaries from the TeX project to provide
some functionality in the form of the ~nocross~ opcode prefix. It
would be nice if we could use off-the-shelf functionality instead of
having to re-implement this as in the C version.

The [[https://crates.io/crates/hyphenation][hyphenation crate]] makes it fairly easy to use a dictionary. It
comes pre-configured with [[https://github.com/tapeinosyne/hyphenation/tree/master/dictionaries][a lot]] of TeX and OpenOffice hyphenation
dictionaries. These come not in their standard form but are encoded
using the bincode format. This encoding happens during the build
process of the hyphenation crate, where all the [[https://github.com/tapeinosyne/hyphenation/tree/master/patterns][pattern files]] in
the ~patterns~ directory are ecoded and stored in the ~dictionaries~
directory.

I added the 3 relevant dictionaries from liblouis, namely
~da-dk-g2.dic~, ~de-g1-core-patterns.dic~ and
~de-g2-core-patterns.dic~ to the patterns folder of hyphenation, added
the files to ~build.rs~ and ~hyphenation_commons/src/language.rs~ and
finally built the hyphenation crate with

#+begin_src shell
  cargo build --features build_dictionaries
#+end_src

The liblouis dictionary files were encoded and I grabed them out of
~target/debug/build/hyphenation-4f7fc3b4af290d85/out/dictionaries~.

You can now load this dictionary and hyphenate words:

#+begin_src rust
  use std::error::Error;

use hyphenation::Load;
use hyphenation::{Hyphenator, Language, Standard};

fn main() -> Result<(), Box<dyn Error>> {
    let path_to_dict = "/path/to/da-g2.standard.bincode";
    let en_us = Standard::from_path(Language::Dutch, path_to_dict)?;

    let hyphenated = en_us.hyphenate("bestemmer");
    println!("Hello, {:?}!", hyphenated);

    Ok(())
}
#+end_src

which results in

#+begin_src shell
cargo run
Hello, Word { text: "bestemmer", breaks: [7] }!
#+end_src

You'll notice that I used the language ~Language::Dutch~. The
language ~DanishGrade2~, that I had added to my local version of the
~hyphenation_commons~ crate, does not exist when I use the
~hyphenation~ crate from crates.io. If I use ~Language::EnglishUS~ it
compiles but complains and tells me the the dictionary is in for the
~Language::Dutch~.

The problem is that the ~hyphenation_commons~ crate converts the list
of languages to an enum that is baked into the build. There does not
seem to be a way to load a dictionary with out the ~Language~ enum.
The bincode seems to contain the language in its serialized data
structure.

At the moment it definitely looks like there is more research needed
as to how we could use the hyphenation crate using our own
dictionaries. Maybe we'll have to rip out the relevant parsing code
from ~hyphenation_commons~ and then provide the hyphenator with a
deserialized version of that.

** Decision Drivers

* ACCEPTED Hand-rolled regexp VM instead of the ~regex~/~regex-automata~ crates
- Deciders :: CE
- Date :: [2026-07-04 Sa]

** Context and Problem Statement

liblouis' ~match~, ~context~, ~correct~, and ~pass2~/~pass3~/~pass4~
opcodes use a regex-like pattern language (character classes,
alternation, quantifiers, negation, boundary anchors). Louis-rs
implements this with its own small VM-based regexp engine
(~translator/regexp.rs~) rather than the well-established, heavily
optimized ~regex~ crate or its lower-level ~regex-automata~ API. Given
how much engineering effort goes into a correct, fast regex engine,
and how much of that effort duplicates what those crates already
provide, is hand-rolling still the right call?

This was already considered once, tersely, in ~TODO.org~: "Instead of
hand-rolling a finite state machine to implement regular expressions
we should use regex_automata. — no, for now keep the hand-rolled
regexp engine, as we need to support effects and negation." This ADR
records the fuller reasoning behind that call, which surfaced again
while implementing the ~^~ boundary attribute and ~_N~ lookback for
~match~/~context~ rules (2026-07-03/04).

There is also a data point from this codebase's own history: an
earlier version of the engine ([[*Regexps duplicate character transitions][Regexps duplicate character
transitions]], superseded) shared character-transition nodes across
multiple rules in one graph, DFA-style. It was abandoned specifically
because liblouis rules aren't just "does this match" — each one
carries its own translation payload with its own precedence, and two
rules whose patterns would share a transition node in a merged graph
have no principled way to decide whose payload wins there. The fix
was giving every rule its own independent instruction stream
(branching via VM ~Split~, not shared graph nodes) rather than a
merged automaton.

** Decision Drivers

- ~correct~/~pass2~/~pass3~/~pass4~ rules can guard on, and mutate,
  numbered *pass-variables* — external, mutable state set by one
  rule's action and checked by a *different* rule's pattern later on
  (~VariableEqual~/~NotVariableEqual~, and *effects*). This isn't
  matching text against a pattern; it's an imperative condition on
  state unrelated to the string being scanned.
- Rule selection is precedence/weight-based across potentially
  thousands of independently-compiled patterns per table, some of
  which must be discovered starting at an earlier cursor position and
  held via ~delayed_translations~/~offset()~ until the cursor catches
  up (built for ~match~'s ~pre~, reused for ~context~'s "before
  bracket" checks and lookback). This doesn't correspond to anything
  in a single compiled pattern object's API.
- ~trace~ (the CLI subcommand showing exactly which rule fired at each
  translation step) depends on an inspectable instruction stream with
  attached payloads, not just a match/no-match answer.
- Negation needs to apply to arbitrary sub-patterns (attribute sets,
  groups, boundary checks), not just single characters or classes.

** Considered Options

1. *Hand-rolled VM* (status quo) — a small instruction set
   (~Char~/~Class~/~Split~/~Jump~/~Capture*~/~VariableEqual~/anchors/…)
   interpreted directly, in the style of Russ Cox's [[https://swtch.com/~rsc/regexp/regexp2.html][Regular Expression
   Matching: the Virtual Machine Approach]].
2. *~regex~ crate* — compile each rule's pattern to a ~Regex~, matching
   with the standard library-grade engine.
3. *~regex-automata~ low-level API* — build a custom VM on top of its
   NFA/DFA primitives, reusing its (well-tested, Unicode-aware)
   automaton construction while keeping our own instruction set for
   the parts it can't express.

** Decision Outcome

Chosen option: *Hand-rolled VM* (option 1).

Neither library option can express pass-variable guards or effects —
these are checks against, and mutations of, state external to the
matched text, which has no representation in the theory of regular
languages either crate's API is built on. Both would need a bespoke
layer wrapped around the library to handle them, at which point the
"pure regex" subset the library actually helps with is a minority of
what these rules need — everything involving variables, effects, or
weight-based multi-rule dispatch would still need our own VM.
Building *two* execution models (library automaton for the regular
part, custom interpreter for the rest) and keeping them in sync
seemed like more complexity, not less, for the parts that matter most.

The ~regex~ crate specifically also does not support arbitrary
lookaround or backreferences (a deliberate trade-off for guaranteed
linear-time matching) — so even the "boundary anchor" work
(~EndAnchor~/~NotEndAnchor~, 2026-07-03) would not have come for free
from it; the same external-state-threading (an ~at_start: bool~
computed by the translation loop, since the regexp only ever sees the
remaining suffix and can't determine its own absolute position) would
still have been necessary regardless of the matching backend.

*** Positive Consequences

- Effects, pass-variables, and weight-based rule selection compose
  naturally with the rest of the translation pipeline, since they're
  just more instruction types in the same VM rather than a bolt-on
  layer around a foreign library.
- ~trace~ gets exact per-rule attribution for free — every instruction
  stream carries its own origin rule and precedence.
- Adding a new liblouis construct (as with ~EndAnchor~/~NotEndAnchor~/
  ~Never~ this week) is a small, local, well-understood change: a new
  ~Regexp~ variant, an ~emit~ arm, and one or two VM instruction cases.

*** Negative Consequences

- Slower than a production-grade regex engine (no SIMD, no DFA
  caching) — ~TODO.org~'s still-open "performance" item (offset
  buckets, shrinking the opcode operand) is a standing acknowledgment
  of this cost, not an oversight.
- We own every correctness bug a mature regex engine would have
  already found and fixed. This session alone surfaced several in our
  own engine: ~negate()~ on ~Concat~/~Either~ not honoring De Morgan's
  laws (pre-existing, still open), an infinite VM loop on a
  zero-width-succeeding quantified body (found and fixed 2026-07-03),
  and a missing negation case for the new end-anchor that would have
  panicked (found in review and fixed the same day).

** Pros and Cons of the Options

*** Hand-rolled VM

- Good, because it handles effects, pass-variables, and weight-based
  precedence natively, with no impedance mismatch
- Good, because ~trace~-level introspection is a natural consequence
  of the design, not an add-on
- Bad, because we maintain regex-engine correctness ourselves,
  including edge cases (negation, infinite loops) a mature crate would
  already have solved
- Bad, because raw matching performance is well below what ~regex~/
  ~regex-automata~ achieve

*** ~regex~ crate

- Good, because it is heavily optimized and extensively tested
- Bad, because it has no notion of external state guards or effects —
  every pass-variable-dependent rule would need custom logic wrapped
  around it anyway
- Bad, because it deliberately excludes lookaround/backreferences, so
  boundary-anchor support still needs the same external-state
  workaround we already built
- Bad, because weight-based selection across many independently
  compiled patterns, and per-rule ~trace~ attribution, aren't things
  its API is shaped for

*** ~regex-automata~ low-level API

- Good, because it reuses well-tested, Unicode-aware NFA/DFA
  construction while allowing more custom control than the ~regex~
  crate's high-level API
- Bad, because pass-variables/effects/weight-based dispatch still need
  a separate custom interpreter layered on top, meaning two execution
  models to keep in sync instead of one
- Bad, because this was the option ~TODO.org~ already named and
  rejected for exactly this reason

* DRAFT Match-rule candidate selection: weight vs. table-definition order
- Deciders :: CE
- Date :: [2026-07-04 Sa]

** Context and Problem Statement

Our engine selects among competing ~match~-rule candidates at a given
position by collecting every rule that matches there and picking the
one with the highest ~weight()~ (total characters checked, including
~pre~/~post~ context — see ~primary.rs~'s ~max_by_key~). liblouis does
something different: ~for_selectRule~ walks a hash bucket of candidate
rules in table-definition order and returns on the *first* one that
validates (~lou_translateString.c~, the ~CTO_Match~ case ends in a
bare ~return;~) — it never considers the rest of the bucket once one
rule succeeds.

These two strategies usually agree, but diverge when a well-scoped
rule and a rule whose ~pre~/~post~ can reach arbitrarily far outward
both match the same position. Concretely, in ~en-ueb-g2.ctb~:

#+begin_example
match !%[^_~] ing - 346          # "ing" at the true end of a word
match %[^_]!%[_1]* in !%[_1]*%[^_] 35   # "in" surrounded by boundary-or-more-letters
#+end_example

For "abandoning", liblouis tries the "ing" rule first (it's defined
earlier), it validates, and liblouis stops there. Our engine instead
discovers *both* candidates — the "in" rule's ~pre~ can walk backward
through the whole of "abandon" (all ordinary letters) to reach a
boundary at the true start of input, giving it a much larger weight
than "ing"'s — and picks "in", producing "abandon" + "in" + "g"
instead of the correct "abandon" + "ing".

This surfaced while fixing an unrelated bug: ~BoundaryMode::AssumeAtStart~
(see ~match_pattern.rs~) was wrongly treating any non-boundary ~pre~
check as automatically failing at position 0, which accidentally
suppressed rules like the "in" one whenever discovered at the true
start of input. Fixing that bug correctly (per liblouis semantics) is
what exposed this pre-existing selection mismatch — it isn't new, it
was just never triggered clearly enough before.

Worth being explicit about what kind of fix this is. liblouis's own
first-match-by-definition-order mechanism is genuinely fragile:
nothing in the rule syntax declares "this beats that rule", it's
implicit in table position (and in which hash bucket a rule lands in,
which table authors don't directly control either) — reordering two
rules, or a table merge that shuffles definitions, can silently
change output with no warning. That's a real liblouis design smell.

Crucially, though, this project's remit is not limited to "replicate
liblouis's behavior, however fragile" — louis-rs's ambition includes
being a *better, more logical* implementation, and its author is
also a liblouis co-maintainer with the standing to change liblouis's
own tables, not just louis-rs's engine. That reopens an option a
strict-compatibility project wouldn't have: fixing the *rule* so it
is unambiguous by construction (e.g. giving the "in" rule an explicit
"not followed by g" exclusion), which fixes both engines at once and
is a more honest expression of what the rule was trying to say in
the first place, rather than reproducing liblouis's order-dependent
tie-break as a permanent feature of louis-rs. See
~feedback_dont_default_to_liblouis_compatibility~ in the assistant's
memory for the fuller reasoning.

That said, upstream table changes have a much bigger blast radius
than an engine-side fix confined to this repo — they affect every
liblouis consumer, not just louis-rs, and go through liblouis's own
review process on its own timeline. Whether and when to pursue that
is the co-maintainer's call, not a default.

** Decision Drivers

- For options 1–3 (the engine keeps behaving the same for a given
  table), matching liblouis's actual output takes priority over how
  "clever" or general our own selection heuristic is — cleverness
  that diverges from liblouis's real behavior is a bug there, not an
  improvement. Option 4 is the deliberate exception: it changes what
  the table itself considers correct, so it isn't judged against
  liblouis's current output at all
- Whatever engine-side fix is chosen must not regress the (large) set
  of cases where weight-based selection already agrees with liblouis
- Precedence/definition-order is already a first-class concept
  elsewhere in the codebase (trie rule precedence, the
  ~backwards_compatibility~ feature flag) — reusing rather than
  duplicating that mechanism would be preferable if it fits here
- Fixing an ambiguous rule upstream is preferable to reproducing its
  ambiguity when it's genuinely a design smell (not just an engine
  gap) and the scope/timing works for liblouis's own release process

** Considered Options

1. *Leave weight-based selection as-is* — accept the mismatch as a
   known gap for the (so far rare) cases it affects.
2. *Definition-order tie-break* — when multiple match candidates
   apply at a position, prefer the one defined earliest in the table,
   falling back to weight only when order is somehow unavailable/tied.
3. *Full first-match-wins* — replicate liblouis's behavior exactly:
   stop at the first candidate (in definition order) that validates,
   never comparing weight at all. Would need care to confirm this
   doesn't regress cases that currently work by picking the
   *longest* of several validating candidates.
4. *Fix the ambiguous rule(s) upstream* — e.g. scope the "in" rule so
   it cannot match the tail of "abandoning" at all, removing the
   collision at the source instead of arbitrating it in either
   engine. Doesn't need to be exclusive of 1–3: even with this
   pursued for known cases, the engine still needs *some* selection
   behavior for tables that haven't been (or won't be) revisited.
   Requires first assessing how common this class of ambiguity is
   across the shipped tables, to judge whether it's a one-off or a
   pattern worth addressing systematically.

   Prototyped for the "in"/"ing" case specifically (not yet committed
   upstream): give "in" a new numbered class excluding a following
   ~g~/~G~, so it can no longer match the start of an "-ing" suffix —

   #+begin_example
   attribute 3 Gg
   match %[^_]!%[_1]* in %[^_]|!%[_13]!%[_1]*%[^_] 35
   #+end_example

   Verified against a real liblouis build: identical output on the
   original collision words plus edge cases (standalone "in" before
   punctuation, "begin", "ring"/"wing"/"bring"/"sing" — the same "in"
   + "g" collision recurs there too), and zero regressions across
   every yaml test exercising ~en-ueb-g2.ctb~ (~112k tests).

   Two liblouis pattern-language traps surfaced while getting this
   right, worth remembering for any future table-level fix: wrapping
   a negated check in ~(...)?~ is a no-op (a failed negation is just
   skipped, and the following token often accepts the same character
   anyway); and ~!(A|B)~ does *not* De Morgan into "matches neither" —
   liblouis's alternation ORs the two negated branches instead of
   ANDing them, so it degenerates to "matches almost anything" whenever
   the two branches are mutually exclusive. The same mistake already
   exists unnoticed in ~afr-za-g2.ctb:229~. Combining an attribute-set
   exclusion with a literal-character exclusion into one real check
   only works by folding both into a single numbered class.

** Decision Outcome

The general engine-side question (options 1–3) is still undecided —
no change has been made to ~match_pattern.rs~ or the candidate
selection logic. A first attempt at fixing the underlying
~BoundaryMode::AssumeAtStart~ bug (option: just let non-boundary
checks match normally, same as ~Otherwise~) is correctly targeted for
that bug in isolation, but caused a net regression on the full test
corpus (94.9% → 94.4% total, 93.9% → 91.8% on
~en-ueb-g2-dictionary_harness.yaml~) once it made rules like the "in"
one above newly reachable at position 0. That fix was reverted rather
than shipped. Revisiting it requires deciding among the options above
first.

Option 4 has been carried out for this one concrete collision (see
prototype above) and left sitting uncommitted in the local liblouis
checkout — pushing it upstream is the co-maintainer's call on timing,
not done as part of this work. It removes the "abandoning" case from
being a live example of the mismatch, but the underlying engine
question remains open for every other table that hasn't been (and may
never be) similarly revisited.
